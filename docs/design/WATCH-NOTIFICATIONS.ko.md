# 워치 알림 & 글랜스 표면 UX 설계 워크시트 (도메인 전문가용)

이 문서 하나로 **"손목과 잠금화면에서 3초 안에 무엇을 보여줄 것인가"**를
결정합니다. 코드는 필요 없습니다 — 당신의 역할은 §3의 설계 질문에 답하고,
그 결정을 §4 양식에 근거·검증 방법과 함께 기록하는 것입니다.

읽는 순서: `docs/EXPERT-ONBOARDING.ko.md`(역할·QA 프로토콜) →
`docs/PLAN.md` §8.5(알림 문법)·§11(알림 소음) → 이 문서.

```
[healthmes 서버]  ──(글랜스 API: 해석된 요약값만)──▶  [폰 위젯 / 워치 / 알림]
     ▲                                                └─ 이 문서가 정하는 것:
     └─ 알림 소음 게이트(쿨다운·예산·방해금지)는                각 표면에 "무엇을,
        서버가 강제 — 표면은 우회 불가                           어떤 말로, 언제"
```

## 1. 고정 제약 — 전문가가 바꿀 수 없는 것

아래는 아키텍처·플랫폼·프로젝트 원칙이 이미 못박은 경계입니다. §3의 모든
답은 이 안에서 움직여야 합니다. (경계 자체가 틀렸다고 판단되면 결정 표에
적지 말고 GitHub 이슈로 이의를 제기해 주세요 — `--repo NomaDamas/healthmes-agent`.)

### 1.1 알림 문법 (PLAN §8.5 — 이것이 디자인 시스템)

모든 선제 메시지는 어떤 표면에서든 같은 문법의 **부분집합**입니다.
표면이 작아지면 줄을 통째로 떨어뜨리는 것만 허용되고, 순서 변경·새 줄
발명은 불가:

```
[관찰 1줄] 오늘 회복 점수 38, 어젯밤 깊은수면 22분.
[근거 1줄] 최근 2주 평균 대비 HRV -18%.
[제안]     14시 집중 블록을 내일 오전으로 옮기고 오후는 가벼운 일만 배치할게요.
[버튼]     ✅ 적용   ✏️ 수정   ❌ 오늘은 그대로      (원탭 — 대화 강요 금지)
[링크]     왜 이 판단? → …/decisions/abc123        (의사결정 뷰어 딥링크)
```

- 축소 표면(컴플리케이션·인라인 위젯)은 관찰 줄의 **압축 표기**만 남습니다.
  압축의 어휘(숫자냐 상태어냐)가 §3 Q3의 결정 대상입니다.
- "왜 이 판단?" 링크는 어떤 표면에서도 제거하지 않습니다 — 탭 가능한
  표면이면 딥링크로, 아니면 상위 표면(알림·타일 탭)으로 위임합니다.

### 1.2 로컬-first 데이터 경로

- 컴패니언 앱·위젯·워치가 소비하는 데이터는 **오직**
  `GET /v1/briefing/glance` (본인 healthmes 인스턴스, base-URL + bearer 토큰
  페어링 — `apps/android-usage/`와 동일한 방식)와 알림 속 의사결정 뷰어
  딥링크뿐입니다. 제3자 클라우드는 데이터 경로에 없습니다.
- 응답은 5분 캐시(`Cache-Control: private, max-age=300`) + ETag 재검증 —
  **표면의 데이터 신선도 하한은 사실상 5분**입니다. "실시간 심박" 같은
  설계는 불가능하며, 시도 대상도 아닙니다.
- APNs/FCM 푸시 릴레이는 없습니다(로컬-first 보류 결정). 따라서 현 단계의
  네이티브 알림은 **폴링에서 파생된 로컬 알림**(지연 수 분~수십 분 가능)이고,
  **즉시성이 필요한 알림의 신뢰 채널은 여전히 Telegram**(워치 미러링 포함)
  입니다. 설계는 이 지연 특성을 전제로 해야 합니다.

### 1.3 플랫폼 예산 (표면이 물리적으로 허용하는 것)

| 플랫폼 제약 | 수치/동작 | 함의 |
|---|---|---|
| iOS WidgetKit 타임라인 갱신 | 시스템 예산제 — 자주 보는 위젯 기준 하루 약 40–70회(≈15–60분 간격), 앱이 강제 갱신 불가 | 위젯은 "지금 값"이 아니라 "최근 값 + 신선도 표시"로 설계 |
| iOS 잠금화면 accessory 계열 | inline = 한 줄(대략 20자 내외에서 잘림, 기기·글꼴 의존 — 실측 필요), circular = 게이지+짧은 값, rectangular = 짧은 3줄 안팎 | 관찰 줄의 압축 표기가 필수 |
| watchOS 컴플리케이션 | WidgetKit accessory 계열(원형/코너/직사각) + 동일 예산제 — 15분보다 촘촘한 갱신은 설계에서 배제 | 첫 화면 지표는 1개 값 + 최소 맥락 |
| Wear OS 컴플리케이션 | `SHORT_TEXT` 텍스트 필드 **최대 7자**(플랫폼 가이드라인), 게이지는 `RANGED_VALUE` | "회복중"(3자)은 들어가고 "회복 중입니다"는 안 들어감 |
| Wear OS 타일 | freshness 간격 기반 갱신, 과도 갱신은 배터리 예산으로 제한 | 폴링 주기와 정합(15분 단위) |
| Android 위젯/백그라운드 | AppWidget 자동갱신 최소 30분, WorkManager 주기 작업 최소 15분 | Android 폴링 주기 = 15분이 바닥 |
| 알림 액션 버튼 | Android 최대 3개(✅✏️❌로 소진), iOS는 관행상 4개 이하 | 문법의 버튼 3종 외 추가 버튼 여유 거의 없음 |
| 워치 알림 미러링 | Apple Watch: 폰 잠금+워치 착용 시 폰 알림이 워치로 미러, 액션 버튼 전달. Wear OS: 기본 브리징(앱이 local-only로 제외 가능), 액션 전달 | 워치 전용 알림 UI를 만들기 전에도 문법 전체가 손목에 도달 — 미러 화면에서의 가독이 1차 설계 대상 |
| 워치 입력 | 카메라 없음, 음성 빠른답장 가능(PLAN §8) | 워치의 응답 수단 = 원탭 버튼 + 음성 |

### 1.4 접근성 (프로젝트 명시 요구 — issue #7)

- **다이나믹 타입/큰 글꼴**: 모든 표면이 시스템 글꼴 크기 확대를 따라야
  하며, 확대 시 어떤 줄부터 떨어뜨릴지(강등 순서)도 설계의 일부입니다.
- **스크린리더**(VoiceOver/TalkBack): 모든 슬롯에 읽기 라벨 — 숫자만
  노출하는 표면도 "인지에너지 58점, 신뢰도 높음"처럼 읽혀야 합니다.
  §3의 표기 결정에는 **낭독 문구**가 함께 필요합니다.
- **고대비**: 색만으로 위험/코칭을 구분하는 설계 금지 — 색+형태(아이콘·
  테두리·위치) 이중 부호화가 기본.

### 1.5 알림 소음 원칙 (PLAN §11 — 서버가 강제, 표면은 우회 불가)

"잘못 울리는 비서는 일주일 안에 음소거된다"가 이 프로젝트의 최대 리스크
정의입니다. 다음 게이트는 `healthmes/engine/triggers.py`가 푸시 **전에**
서버에서 강제하며, 어떤 표면·앱 설정으로도 우회할 수 없습니다:

| 게이트 | 기본값 (env로 조정 가능) | 동작 |
|---|---|---|
| 방해금지(quiet hours) | 22:30–07:00 (사용자 로컬 시간) | 창 안에서는 푸시 자체가 없음 |
| 룰별 쿨다운 | 60분 | 같은 룰의 재푸시 최소 간격 |
| 일일 예산 | 하루 8건 | 초과분은 푸시하지 않음 |

- 억제된 발화는 **기록만 되고**(`alert_sent=false` + 억제 사유) 푸시되지
  않으며, 현재 글랜스 API의 `alerts` 블록에도 나타나지 않습니다 — "밤새
  억제된 이벤트를 아침 표면에 어떻게 보여줄 것인가"가 §3 Q6의 핵심입니다.
- 전문가가 정하는 것은 게이트의 존재가 아니라 **게이트 통과 후의 표현**
  (진동 세기, 시각 등급, 야간 이월 표기)과, 실측 근거가 쌓인 뒤의
  **기본값 튜닝 제안**(이슈로)입니다.

## 2. 설계 대상 표면 인벤토리 & 데이터 슬롯

### 2.1 데이터 원천 — 글랜스 API 계약 (구현 완료, `healthmes/api/briefing.py`)

모든 표면이 읽는 단일 페이로드. **여기 없는 데이터는 표면에 못 올립니다**
(새 필드가 필요하면 §4 결정 표에 "요구 필드"로 기록 — 엔지니어링이 받음):

```jsonc
GET /v1/briefing/glance        // bearer 토큰, 5분 캐시 + ETag
{
  "generated_at": "2026-07-10T05:23:00Z",
  "timezone": "Asia/Seoul",
  "energy": {
    "score": 58,               // 0–100 정수 | null(정직한 결측)
    "confidence": "high",      // high|medium|low — 최신 값의 신선도 사다리
    "curve_24h": [ {"hour": 0, "score": null}, …24개… ]   // 오늘 로컬 하루
  },
  "next_blocks": [             // 최대 3개, 임박순
    { "start": "…", "end": "…", "title": "Deep work",
      "energy_demand": "low"|"med"|"high"|null,
      "source": "calendar"|"proposal" }
  ],
  "alerts": {
    "unresolved_count": 2,     // 최근 24h 내 푸시된 알림 수(placeholder 정책)
    "top": { "rule_id": "stress_spike_vs_baseline",
             "summary": "…관찰 1줄…",
             "decision_url": "…/decisions/…" }            // | null
  },
  "latest_decision": { "id": "…", "url": "…" }            // | null
}
```

알림(푸시) 쪽 원천은 트리거 이벤트이며, 현재 룰은 4종입니다 — Q2(위험 vs
코칭)의 분류 대상: `stress_spike_vs_baseline`, `low_recovery_heavy_afternoon`,
`schedule_changed`, `deadline_risk`. 각 발화는 관찰(`summary`)·근거(`evidence`)·
제안(`proposal`) 줄을 이미 갖고 있습니다.

### 2.2 표면별 슬롯 표 (플레이스홀더 현황 = 시작점, 최종안은 전문가 결정)

"슬롯"은 그 표면이 물리적으로 수용하는 정보 자리입니다. 각 슬롯에 어떤
필드를 어떤 표기로 넣을지가 설계입니다.

**폰 잠금화면 위젯 — 3계열 (iOS accessory; Android는 알림 존재감으로 대응)**

| 계열 | 슬롯 예산 | 플레이스홀더 매핑 (현 구현) | 설계 여지 |
|---|---|---|---|
| inline (한 줄) | 텍스트 ~20자 | `energy.score` + `next_blocks[0].start` ("E58 · 14:00 집중") | 압축 표기법(Q3), 결측/저신뢰 표기(Q5) |
| circular (원형) | 게이지 1 + 중앙 값 1 | `energy.score` 숫자 + 0–100 게이지 | 게이지 의미(점수 vs 하루 진행), 색 등급(Q2·Q3) |
| rectangular (3줄) | 짧은 3줄 | 1줄 `energy.score`+`confidence` / 2줄 `next_blocks[0]` / 3줄 `alerts.top.summary` | 줄 우선순위와 강등 순서(Q1), 위험 표시(Q2) |

**폰 홈 위젯 — 2크기 (iOS small·medium ≈ Android 2×2·4×2)**

| 크기 | 슬롯 예산 | 플레이스홀더 매핑 | 설계 여지 |
|---|---|---|---|
| small | 값 1 + 보조줄 1 | `energy.score`(대형) + `next_blocks[0].title`·`start` | 무엇이 "대형 1값"인가(Q1과 동형) |
| medium | 값 1 + 스파크라인 + 리스트 2 + 뱃지 | `energy.score` + `curve_24h` 스파크라인 + `next_blocks[0..1]` + `alerts.unresolved_count` 뱃지 | 커브의 결측(null) 구간 표현(Q5), 블록의 `energy_demand` 노출 여부 |

**워치 컴플리케이션 — 원형 / 코너 / 직사각**

| 계열 | 슬롯 예산 | 플레이스홀더 매핑 | 설계 여지 |
|---|---|---|---|
| 원형(circular) | 값 1 (+미니 게이지) | `energy.score` | **Q1의 본진**: 이 한 자리에 올릴 지표 |
| 코너(corner) | 값 1 + 곡선 라벨 | `energy.score` + "energy" 라벨 | 라벨 어휘(Q3), Wear `SHORT_TEXT` 7자 공유 |
| 직사각(rectangular) | 2–3줄 | `energy.score`+`confidence` / `next_blocks[0]` / `alerts.unresolved_count` | 손목에서의 줄 우선순위(Q1), 위험 구분(Q2) |

탭 동작(고정 플럼빙): 컴플리케이션 탭 → 워치 브리핑 뷰(에너지 커브 +
다가오는 블록 3개 + 최근 알림 1개, 크라운 스크롤). 현재 플레이스홀더
랜딩 — watchOS: 워치 앱 홈(에너지·알림·다음 블록 텍스트), Wear OS:
타일·컴플리케이션 탭 모두 온워치 페어링/상태 화면. 괄호 안 구성(커브 +
블록 3 + 크라운 스크롤)은 설계 목표입니다.

**Wear OS 타일 (스와이프 1장)**

| 슬롯 | 플레이스홀더 매핑 | 설계 여지 |
|---|---|---|
| 헤드 값 | `energy.score` + `confidence` | 표기(Q3)·저신뢰(Q5) |
| 중단 | `next_blocks[0]` (시각·제목) | 블록 없음(null)일 때의 문구 |
| 하단 | `alerts.top.summary` 1줄 또는 `unresolved_count` | 위험 강조(Q2), 탭 → 폰 딥링크 vs 워치 뷰 |

**알림 — 축소 / 확장 / 워치 미러 (문법의 본체)**

| 형태 | 슬롯 예산 | 플레이스홀더 매핑 | 설계 여지 |
|---|---|---|---|
| 축소(배너/잠금화면) | 제목 1 + 본문 1–2줄 | 제목=룰 한글명, 본문=관찰 줄(`summary`) | 제목 어휘·아이콘 등급(Q2) |
| 확장(long-press/펼침) | 본문 여러 줄 + 버튼 ≤3(Android)/≤4(iOS) | 관찰/근거/제안 3줄 + ✅✏️❌ + "왜?" 딥링크(`decision_url`) | 버튼 문구, 근거 줄 생략 조건(Q5) |
| 워치 미러 | 스크롤 카드 + 액션 + 음성답장 | 문법 전체가 그대로 미러됨 | 손목 1스크린에 들어올 줄 수, 진동 등급(Q4), 야간(Q6) |

## 3. 전문가가 답할 설계 질문

각 질문 아래 "현재 플레이스홀더"는 지금 코드가 임시로 하는 동작입니다 —
동의하면 그대로 승인 결정으로 기록해도 됩니다.

**Q1. 워치 첫 화면(컴플리케이션 1값)에 올릴 가치가 있는 지표는 무엇인가?**
후보: 인지에너지 점수(현 플레이스홀더), 다음 블록까지 남은 시간, 미확인
알림 수, 혹은 시간대별 전환(오전엔 에너지, 저녁엔 내일 첫 블록). 손목
글랜스의 기회비용(다른 앱 컴플리케이션과의 경쟁)까지 고려해 주세요.
- 현재 플레이스홀더: 항상 `energy.score`.

**Q2. 위험성 알림과 코칭 알림을 시각적으로 어떻게 구분하는가?**
현 룰 4종을 등급으로 분류하고(예: `stress_spike_vs_baseline`은 위험?
`deadline_risk`는 코칭?), 등급별 색+형태 이중 부호화(§1.4 고대비)와
표면별 표현(컴플리케이션 뱃지, 알림 아이콘/제목 접두)을 정해 주세요.
등급 체계가 서버 필드로 필요하면(예: `alerts.top.severity`) 요구 필드로
기록하세요.
- 현재 플레이스홀더: 등급 구분 없음, 모든 알림 동일 취급.

**Q3. 숫자(58) vs 상태어("회복중"/"주의") — 어느 표면에 어느 표기인가?**
점수의 임상적 해상도(58과 62의 차이가 사용자 행동을 바꾸는가), 상태어의
구간 정의(몇 점부터 "주의"인가), Wear `SHORT_TEXT` 7자 예산, 스크린리더
낭독 문구까지 한 세트로 정해 주세요. 상태어 구간은 서버 결정론 계산으로
옮길 수 있습니다(요구 필드: 예 `energy.state_label`).
- 현재 플레이스홀더: 모든 표면 숫자 원값.

**Q4. 알림 등급별 진동(햅틱) 정책은?**
손목 진동은 가장 침습적인 채널입니다. Q2의 등급별로 진동 세기/패턴/무음
여부를, 그리고 §1.5의 서버 게이트를 통과한 알림이라도 워치에서는 무음
표시만 할 조건이 있는지 정해 주세요. (채널 중요도는 OS 알림 채널/카테고리
설정으로 구현됩니다.)
- 현재 플레이스홀더: OS 기본 진동, 등급 무차별.

**Q5. 저신뢰(`confidence: low`) 및 결측(null) 표시 규칙은?**
글랜스 API는 신선도 사다리(현재 시간대 값=high, ≤3h 경과=medium, 그 외/
무데이터=low)와 정직한 null(커브의 결측 시간대, 점수 없음)을 돌려줍니다.
low일 때 숫자를 그대로 보여줄지(오신뢰 위험), 흐림/물음표/생략 중 무엇으로
할지, 확장 알림에서 근거 줄을 생략할 조건("confidence 낮으면 단정 금지" —
온보딩 §1의 의무)을 표면별로 정해 주세요.
- 현재 플레이스홀더: confidence 텍스트 병기, null은 빈 칸.

**Q6. 야간/방해금지 시간의 표면 동작은?**
서버가 22:30–07:00에 푸시를 억제하는 것은 고정입니다(§1.5). 정할 것:
① 그 시간대 위젯/컴플리케이션이 낮 데이터를 계속 보여줄지, 야간 모드
(내일 첫 블록·수면 유도 문구)로 바뀔지 ② 밤새 억제·기록된 이벤트를 아침
표면(위젯 뱃지? 아침 브리핑 병합?)에 어떻게 이월할지 — 현재는 글랜스
`alerts`에 아예 나타나지 않습니다 ③ 아침 07:00 직후 몰아치는 재개 알림의
상한. 필요하면 요구 필드(예: `alerts.suppressed_overnight_count`)로.
- 현재 플레이스홀더: 시간대 무관 동일 렌더링, 억제 이벤트 미노출.

## 5. 현재 구현 상태 — 정직 고지

이 문서의 결정이 곧 다음 구현 스펙입니다. 지금 있는 것과 없는 것:

**구현되어 있음 (안정 계약)**
- `GET /v1/briefing/glance` (`healthmes/api/briefing.py`) — §2.1의 필드
  전부, 5분 캐시/ETag, bearer 인증. 값은 전부 저장된 결정론 계산의 재노출
  (위젯 폴링이 웨어러블 API를 건드리지 않음).
- 서버 측 알림 위생 게이트(§1.5)와 트리거 4종, 알림 문법 줄
  (관찰/근거/제안)의 생성.
- Telegram 채널 전체(워치 미러 포함) — 현행 신뢰 푸시 경로.

**플럼빙+플레이스홀더만 있음 (최종 UX 아님 — §2.2 표의 "플레이스홀더
매핑"이 그 전부)**
- 이 wave에서 추가되는 iOS/Android/워치 컴패니언 스캐폴드(`apps/` 하위):
  페어링, 글랜스 폴링, 각 표면의 자리표시 렌더링까지. 표기·색·등급·진동은
  전부 임시값입니다.

**아직 없음 / 보류 (결정 대기 또는 후속 플럼빙)**
- 알림의 근거(`evidence`)·제안(`proposal`) 줄은 트리거 페이로드에는 있지만
  글랜스 API `alerts.top`에는 `summary`(관찰)만 노출 — 확장 알림을 폴링으로
  온전히 렌더하려면 노출 결정 필요(Q2·Q5와 연동).
- 알림 "해결됨" 추적 없음 — `unresolved_count`는 "최근 24시간 내 푸시됨"의
  자리표시 정책입니다(개선 대상으로 명시됨).
- 야간 억제 이벤트의 표면 이월 없음(Q6), 알림 등급 필드 없음(Q2),
  상태어 필드 없음(Q3).
- APNs/FCM 릴레이 없음(§1.2), Live Activities/Dynamic Island 미착수,
  Wear OS Ongoing Activity(현재 집중 블록 상시 표시 — issue #7 Wear 범위
  항목) 미착수, 워치 음성 빠른답장의 캡처 라우팅 미착수.
